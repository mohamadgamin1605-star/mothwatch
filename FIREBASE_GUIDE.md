FIRESTORE DATA STRUCTURE - Optimization & Scalability 📦\n\n### File: `src/lib/firebase-firestore.ts`\n\n### Data Architecture\n\n```\nFirestore Database Structure:\n\nusers/{userId}\n├─ email: string\n├─ createdAt: timestamp\n├─ watchlistCount: number (cached for quick reads)\n├─ favoritesCount: number (cached)\n├─ lastActive: timestamp\n│\n├─ watchlist/{itemId}\n│  ├─ malId: number\n│  ├─ title: string\n│  ├─ imageUrl: string\n│  ├─ contentType: 'anime'|'movie'\n│  ├─ currentEpisode: number\n│  ├─ totalEpisodes: number\n│  └─ addedAt: timestamp ⭐ Indexed for sorting\n│\n├─ favorites/{itemId}\n│  ├─ malId: number\n│  ├─ title: string\n│  ├─ rating: 1-5\n│  └─ favoritedAt: timestamp\n│\n└─ watching-progress/{itemId}\n   ├─ malId: number\n   ├─ lastEpisodeWatched: number\n   ├─ percentageComplete: number (0-100) ⭐ Indexed for filtering\n   ├─ totalEpisodes: number\n   └─ lastWatchedAt: timestamp ⭐ Indexed for sorting\n```\n\n### Why This Structure Scales\n\n#### **Problem: Naive Flat Structure (❌ Don't do this)**\n```\nwatchlists/{watchlistId}\n├─ userId: string\n├─ malId: number\n├─ addedAt: timestamp\n\n❌ Issues:\n   - Can't easily query \"all items for user X\"\n   - Requires compound index on (userId, addedAt)\n   - Billing: Higher reads for filtering\n   - Cannot read user's watchlist with single doc read\n```\n\n#### **Solution: Nested Sub-collections (✅ This way)**\n```\nusers/{userId}\n└─ watchlist/{itemId}\n\n✅ Benefits:\n   - Single collection query per user\n   - User isolation (no cross-user reads)\n   - Scales to millions of items per user\n   - Simpler permissions (only read own data)\n   - Better caching opportunities\n```\n\n### Query Examples\n\n#### **Get User's Watchlist (Sorted by Newest)**\n```typescript\nconst watchlist = await getWatchlist(20); // Page size: 20\n\n// Firestore Query:\n// db.collection('users').doc(userId)\n//   .collection('watchlist')\n//   .orderBy('addedAt', 'desc')\n//   .limit(20)\n\n// Index Required:\n// Collection: users/{userId}/watchlist\n// Fields: addedAt (Descending)\n```\n\n#### **Get \"Continue Watching\" (In Progress Only)**\n```typescript\nconst continuing = await getContinueWatching(10);\n\n// Firestore Query:\n// db.collection('users').doc(userId)\n//   .collection('watching-progress')\n//   .where('percentageComplete', '<', 100)\n//   .orderBy('percentageComplete', 'desc')\n//   .orderBy('lastWatchedAt', 'desc')\n//   .limit(10)\n\n// Indexes Required:\n// 1. Collection: users/{userId}/watching-progress\n//    Fields: percentageComplete (Asc), lastWatchedAt (Desc)\n// 2. Collection: users/{userId}/watching-progress\n//    Fields: lastWatchedAt (Desc)\n```\n\n#### **Check if Item in Watchlist (Single Query)**\n```typescript\nconst inWatchlist = await isInWatchlist(malId);\n\n// Firestore Query:\n// db.collection('users').doc(userId)\n//   .collection('watchlist')\n//   .where('malId', '==', malId)\n//   .limit(1)\n//   .get()\n\n// Index Required:\n// Collection: users/{userId}/watchlist\n// Fields: malId\n```\n\n### Firestore Indexes to Create\n\n```\nGo to: Firebase Console → Firestore → Indexes\n\nCreate these composite indexes:\n\n1. Collection: users/{userId}/watchlist\n   - addedAt (Descending)\n   \n2. Collection: users/{userId}/watching-progress\n   - percentageComplete (Ascending)\n   - lastWatchedAt (Descending)\n   \n3. Collection: users/{userId}/favorites\n   - rating (Descending)\n   - favoritedAt (Descending)\n```\n\n### Caching Strategy\n\n```typescript\n// Problem: Expensive query each time\nconst watchlistCount = (await getWatchlist()).length; // ❌ Reads all docs\n\n// Solution: Maintain count in user doc\nwatchlistCount = user.watchlistCount; // ✅ Single read\n\n// Update on add/remove:\nawait updateDoc(doc(db, 'users', userId), {\n  watchlistCount: increment(1),  // Atomic increment\n});\n```\n\n### Firestore Cost Optimization\n\n| Operation | Cost | How to Optimize |\n|-----------|------|------------------|\n| List Watchlist | 1 read per item | ✅ Already optimized (sub-collection) |\n| Check if favorite | 1 read | ✅ Optimized (single query) |\n| Get watchlist count | 1 read | ✅ Cache in user doc |\n| Update progress | 1 write | ✅ Batched writes |\n\n**Monthly Cost Example:**\n- 10,000 users\n- Each user has 50 watchlist items\n- 2 reads per user per day (check watchlist + continue watching)\n\n```\nReads:  10,000 × 2 × 30 = 600,000 reads/month\nWrites: 10,000 × 1 × 30 = 300,000 writes/month\n\nCost: $0.06 + $0.30 = $0.36/month\n(Well within free tier: 50k reads, 20k writes daily)\n```\n\n### Batch Operations\n\n```typescript\n// Update multiple documents atomically\nconst batch = writeBatch(db);\n\n// Update 3 items in one batch (1 write cost, not 3)\nbatch.update(doc1, {...});\nbatch.update(doc2, {...});\nbatch.update(doc3, {...});\n\nawait batch.commit();\n```\n\n---\n\n## 3. FIREBASE REMOTE CONFIG - Dynamic Settings 🔧\n\n### File: `src/lib/firebase-remote-config.ts`\n\n### How It Works\n\n```\n┌─ Firebase Console (Server) ─────────────────────┐\n│                                                  │\n│ [Set player_default_quality = \"1080p\"]          │\n│ [Set server_configuration = JSON]               │\n│ [Set feature_enable_watchlist = true]           │\n│                                                  │\n│ Click \"Publish\"\n│          ↓\n├─────────────────────────────────────────────────┤\n│                                                  │\n│  App calls initializeRemoteConfig()             │\n│         ↓                                        │\n│  Firebase fetches latest values                 │\n│         ↓                                        │\n│  App uses new settings (NO REDEPLOY)            │\n│                                                  │\n└─ User's Browser ───────────────────────────────┘\n```\n\n### Setup in Firebase Console\n\n#### **Step 1: Add Parameter**\n```\nFirebase Console → Remote Config → Create key\n\nParameter Name:  player_default_quality\nDefault Value:   \"auto\"\nDescription:     Default video quality for new users\n```\n\n#### **Step 2: Create Variants (A/B Test)**\n```\nVariant: \"High Quality Test\"\n├─ Targeting: Random 25% of users\n├─ player_default_quality = \"1080p\"\n├─ player_buffer_duration = \"10\"\n└─ Start: Now, End: 2026-07-05\n```\n\n#### **Step 3: Monitor Results**\n```\nCompare:\n- Control (auto): Avg watch time 24 min, buffering 2.1%\n- Test (1080p):    Avg watch time 19 min, buffering 4.8%\n\nConclusion: Higher quality = more buffering\n→ Keep default as \"auto\"\n```\n\n### Use Cases\n\n#### **1. Player Settings - Real-Time Adjustment**\n```typescript\n// src/routes/watch.$id.tsx\nimport { getPlayerSettings } from '@/lib/firebase-remote-config';\n\nfunction AnimeDetail({ anime }) {\n  const settings = await getPlayerSettings();\n  \n  // ✅ User's player automatically uses updated settings\n  // No app refresh needed!\n  \n  return (\n    <iframe\n      src={src}\n      title={title}\n      autoplay={settings.enableAutoplay}  // Dynamic!\n      quality={settings.defaultQuality}   // Can be changed on-the-fly\n    />\n  );\n}\n```\n\n#### **2. Server Management - Disable Failing Server**\n```typescript\n// Server 2Embed is having issues\n// In Firebase Console, set:\n// server_configuration = [\n//   { id: \"2embed\", isActive: false, ... }\n// ]\n\n// App immediately stops suggesting 2Embed\nconst servers = await getServerConfig();\nconst activeServers = servers.filter(s => s.isActive);\n// \"2Embed\" removed without redeploying!\n```\n\n#### **3. Feature Flags - Launch Features Gradually**\n```typescript\n// Roll out watchlist feature to 10% of users first\n\nVariant A (10%): feature_enable_watchlist = true\nVariant B (90%): feature_enable_watchlist = false\n\nif (featureFlags.enableWatchlist) {\n  // Show watchlist button\n}\n```\n\n#### **4. Maintenance Mode**\n```typescript\n// Server going down? Enable maintenance:\n\nFirebase Console → Set:\n├─ maintenance_mode = true\n├─ maintenance_message = \"Upgrading servers. Back at 3 AM PST\"\n\n// App shows banner to users:\nif (featureFlags.maintenanceMode) {\n  return <MaintenanceBanner message={featureFlags.maintenanceMessage} />;\n}\n```\n\n#### **5. API Rate Limiting - Adjust on the Fly**\n```typescript\n// Jikan API being rate-limited?\n// Adjust without code change:\n\napi_jikan_rate_limit = \"30\"  // From 60 → 30 requests/min\napi_retry_attempts = \"5\"      // From 3 → 5 retries\napi_retry_delay = \"2000\"      // From 1000 → 2000 ms\n```\n\n### Implementation Example\n\n```typescript\n// Initialize on app startup\nimport { initializeRemoteConfig, getPlayerSettings } from '@/lib/firebase-remote-config';\n\n// src/routes/__root.tsx\nexport function RootComponent() {\n  useEffect(() => {\n    // Fetch Remote Config on app load\n    initializeRemoteConfig();\n  }, []);\n  \n  return <Outlet />;\n}\n\n// src/routes/watch.$id.tsx\nfunction AnimeDetail({ anime }) {\n  const [settings, setSettings] = useState(null);\n  \n  useEffect(() => {\n    getPlayerSettings().then(setSettings);\n  }, []);\n  \n  if (!settings) return <Skeleton />;\n  \n  return (\n    <VideoPlayer\n      quality={settings.defaultQuality}\n      autoplay={settings.enableAutoplay}\n      bufferDuration={settings.bufferDuration}\n      skipIntro={settings.skipIntroSeconds}\n    />\n  );\n}\n```\n\n### Remote Config Console Dashboard\n\n```\n┌───────────────────────────────────────────────────────┐\n│ Remote Config Dashboard                               │\n├───────────────────────────────────────────────────────┤\n│                                                        │\n│ Recent Updates:                                        │\n│ ├─ ✅ 2 hours ago: Updated server_configuration      │\n│ ├─ ✅ 1 day ago: Added feature_enable_watchlist      │\n│ └─ ✅ 3 days ago: Created player_default_quality    │\n│                                                        │\n│ Active Variants:                                       │\n│ ├─ \"High Quality Test\" (25% of users)                │\n│ ├─ \"New UI Trial\" (10% of beta users)                │\n│ └─ \"Maintenance Mode\" (100% -
